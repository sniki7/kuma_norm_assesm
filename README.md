# kuma_norm_assesm
Примеры SQL запросов для проверки ошибок нормализации

Поле Extra содержит необработанные парсером данные (например, дополнительные пары «ключ-значение» из исходного события). Если таких полей много, это может указывать на неполный парсер или бесполезно расходовать место в хранилище.
Запрос для поиска событий с большим количеством полей в Extra (рекомендуемый период - 24 часа):
extra_check.sql

В поле FileHash или OldFileHash могут попадать данные, не являющиеся хэшем (MD5, SHA1/256/512 как самые популярные). Из-за этого могут некорректно работать фиды Cybertrace
Запрос для поиска некорректных хэшей (рекомендуемый период — 24 часа):
hash_check.sql

Убедитесь, что в полях, содержащих имена хостов (например, SourceHostName, DestinationHostName), нет недопустимых символов. За основу можно взять соглашения Microsoft с небольшими доработками.
Запрос для поиска некорректных символов в именах хостов (рекомендуемый период — 24 часа):
hostname_check.sql

С помощью функций ClickHouse для работы с IP-адресами можно найти случаи, когда в поле для имени хоста записан IP-адрес. Это может мешать обогащению (например, обратному DNS-поиску).
Запрос, рекомендуемый период — 24 часа:
iphostname.sql

Если в сети используются IPv4-адреса, но в полях SourceHostName/DestinationHostName они представлены в IPv6-формате (например, ::ffff:10.1.1.1), это может затруднить анализ.
Запрос для обнаружения таких адресов (рекомендуемый период — 24 часа). Рекомендуется сохранять оригинальный адрес в отдельное поле, а в основное помещать чистый IPv4:
ipv4ipv6.sql

Использование адресов типа 127.0.0.1 или localhost в поле DeviceAddress может блокировать функции обогащения в KUMA.
Запрос для поиска таких значений (рекомендуемый период — 24 часа). При обнаружении в нормализаторе стоит заменить локальный адрес на пустую строку (''):
localhostinaddr.sql

Вы можете найти события, в которых присутствуют сетевые атрибуты (например, порты или протокол), но отсутствует IP-адрес источника или назначения
Запрос, рекомендуемый период — 24 часа:
network_full.sql

Диапазон портов от 1 до 65535
Запрос для обнаружения таких адресов (рекомендуемый период — 24 часа). Рекомендуется сохранять оригинальный адрес в отдельное поле, а в основное помещать чистый IPv4:
port_check.sql

Полезно отслеживать количество Raw-событий в ежедневной динамике. Резкий рост может сигнализировать о сбое в работе источника или обновлении с несовместимыми изменениями формата.
Запрос, сравнивающий поток событий с непустым полем Raw за вчера и сегодня (рекомендуемый период — 2 дня):
raw_dynamics.sql

В поле RequestUrl могут попадать данные, не соответствующие формату URL. Это сделает неработоспособными правила корреляции, которые ищут индикаторы компрометации типа URL.
Запрос для поиска некорректных URL (рекомендуемый период — 24 часа):
rurl_check.sql

Поля нормализованной модели имеют ограничения по длине. События, превышающие лимит, обрезаются, что ведет к потере данных.
Запрос для полей с ограничением в 1023 символа (рекомендуемый период — 24 часа):
size_1k.sql

Поля нормализованной модели имеют ограничения по длине. События, превышающие лимит, обрезаются, что ведет к потере данных.
Запрос для полей с ограничением в 4000 символов (рекомендуемый период — 24 часа):

Большая разница (например, более суток) между временем на источнике (Timestamp) и временем получения событий SIEM (DeviceReceiptTime или EndTime) указывает на проблемы в инфраструктуре: некорректное время на источнике или задержки при передаче логов.
Запрос для поиска таких аномалий (рекомендуемый период — 24 часа):
timecheck.sql

Запрос для проверки заполнения этих полей (рекомендуемый период — 24 часа):
vendor_check.sql
